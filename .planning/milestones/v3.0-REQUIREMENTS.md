## v3.0 Requirements (All Complete)

### Multi-Source Ingestion (INGEST)

- [x] **INGEST-01**: User can upload multiple ad platform CSVs (Google Ads, Meta Ads, LinkedIn, or custom), assigning a platform name and currency to each upload before running analysis
  - *Outcome: Validated* — 4-platform upload grid in app.py; Load Demo Data sidebar button; 09-VALIDATION compliant
- [x] **INGEST-02**: User can map their CRM CSV's columns to standard fields (session_id, lead_status, projected_value, sales_notes) using an auto-suggest UI that proposes matches and requires explicit confirmation before the merge proceeds
  - *Outcome: Validated* — `auto_suggest_crm_columns()` via difflib; Confirm Mapping gate at app.py:247
- [x] **INGEST-03**: System normalizes all ad platform cost columns to USD using a static FX_RATES dictionary before any aggregation, so cross-platform spend totals are financially meaningful
  - *Outcome: Validated* — `ingest.py::convert_cost()` + `FX_RATES` dict (17 currencies); 17 FX tests green

### Richer Agent Logic (AGENT)

- [x] **AGENT-01**: User receives AI budget recommendations that reference cross-platform comparisons (e.g., which platform delivers better lead quality per dollar spent), based on the unified multi-source dataset
  - *Outcome: Validated* — SYSTEM_PROMPT cross-platform rules; multi-source campaign_agg.to_csv() fed to gpt-4o
- [x] **AGENT-02**: Each campaign action in the results includes a `source_platforms` list identifying which ad platforms contributed sessions to that campaign
  - *Outcome: Validated* — Required Pydantic field in CampaignAction; fixture validated; test_llm.py schema tests green
- [x] **AGENT-03**: System token-counts the aggregated payload with tiktoken before every gpt-4o call and warns the user if the payload approaches 60k tokens, preventing context window overflow
  - *Outcome: Validated* — `count_prompt_tokens()` using o200k_base encoding; token gate at app.py:378; st.warning + "Continue anyway" gate

### Dynamic Views (VIEW)

- [x] **VIEW-01**: User can view a spend vs qualified leads scatter chart (color-coded by platform) and an action distribution chart for the current analysis, presented in a tabbed layout alongside the data preview and AI recommendations
  - *Outcome: Validated* — `st.tabs(["Data Preview","Charts","Campaign Actions"])` at app.py:431; px.scatter + px.bar
- [x] **VIEW-02**: User can filter results by platform, action type, and campaign name, and sort results by spend, campaign name, or recommended action
  - *Outcome: Validated* — 5 widget keys; filter list comprehensions on `result.campaigns`; reset via del + st.rerun
- [x] **VIEW-03**: User can select up to 3 campaigns for side-by-side comparison and drill down to session-level rows for any campaign in the results view
  - *Outcome: Validated* — `disabled=not can_add` max-3 enforcement; Side-by-Side Comparison section; inline session_rows dataframe

### Analysis Management (MGMT)

- [x] **MGMT-01**: User can save the current analysis to Postgres with a label, and reload any past saved analysis from a sidebar list without re-uploading CSV files
  - *Outcome: Validated* — `st_db.save_analysis()` + `st_db.load_analysis()` + `st_db.list_analyses()`; Save Analysis button in Campaign Actions tab; Past Analyses sidebar; AnalysisResult.model_validate() restores session state
- [x] **MGMT-02**: User can download the current analysis as a PDF report (via fpdf2) and as a CSV file, both via st.download_button
  - *Outcome: Validated* — `generate_pdf(result, meta)` → st.download_button; `campaign_agg.to_csv()` → st.download_button
- [x] **MGMT-03**: When the OpenAI API is unavailable, the app falls back to a cached fixture response (DEMO_MODE) so the demo remains functional offline
  - *Outcome: Validated* — `try/except openai.OpenAIError` → `_load_fixture()` fallback in `run_analysis()`; `api_fallback_active` warning banner; startup guard for `init_db()` prevents crash on DB unavailability

## Future Requirements (deferred from v3.0)

- Live exchange rate API integration (static FX_RATES sufficient for v3.0)
- Real Meta/Google/LinkedIn OAuth API connectors (CSV-first until multi-source value is proven)
- More than 5 simultaneous platform uploads (context window degrades beyond this)
- Branded PDF templates and scheduled/recurring analysis
- Shared team workspaces and multi-tenant isolation

---

## Out of Scope

- OAuth / direct API integrations with ad platforms — CSV-first this milestone; full API wiring is v4+
- Real user auth / login / accounts — waitlist-only; auth adds 2–3 phases
- Stripe / billing — no paid tier yet
- Redis job queue / async background jobs — synchronous calls sufficient at current scale
- HubSpot / Salesforce direct API pull — generic CSV covers current needs
