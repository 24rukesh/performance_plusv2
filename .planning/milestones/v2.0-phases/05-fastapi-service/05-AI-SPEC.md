## 1. System Classification

**System Type:** Structured Extraction

**Description:**
The Performance Plus AI system receives tabular web analytics + CRM data as input, sends it as CSV to gpt-4o via `client.beta.chat.completions.parse()`, and returns a structured JSON response of per-campaign budget action recommendations. The system acts as an "Autonomous CMO" — it reads both quantitative click data and qualitative sales-rep notes and produces opinionated, signed budget decisions (increase/pause/decrease by %). Phase 5 exposes this existing logic via FastAPI endpoints. No new AI architecture is introduced; the intelligence layer is the `run_analysis()` + `_call_llm()` path already shipped in `llm.py`.

**Critical Failure Modes:**
1. **Budget action mismatch** — AI recommends `increase` for a campaign where all CRM notes are "Disqualified" / negative sentiment. This wastes ad spend and erodes trust with the first API consumer.
2. **Dropped campaigns** — `AnalysisResult.campaigns` omits one or more input campaigns. An n8n automation routing budget on this output will silently skip those campaigns with no error signal.
3. **Hallucinated evidence_count** — AI returns an `evidence_count` that doesn't match the `session_count` from the aggregated input. Breaks the traceability guarantee that makes the output auditable.

## 1b. Domain Context

> Grounded in practitioner knowledge of performance marketing / adtech operations.

**Industry Vertical:** Performance Marketing / Marketing Operations (AdTech)

**User Population:** Marketing operations teams, performance marketers, agency media buyers, CMOs at SMB/mid-market companies who run paid campaigns on Meta/Google. API consumers include n8n automation workflows and CRM webhook senders that act on budget decisions programmatically.

**Stakes Level:** Medium

**Output Consequence:** API consumers (n8n, CRM webhooks) receive budget action recommendations and are expected to act on them — routing spend increases, pausing underperformers, or flagging campaigns for human review. A bad recommendation that reaches an automation workflow results in misallocated ad budget. The v2.0 API is single-tenant and used by informed operators, reducing but not eliminating risk.

### What Domain Experts Evaluate Against

| Dimension | Good (expert accepts) | Bad (expert flags) | Stakes | Source |
|-----------|----------------------|--------------------|--------|--------|
| **Budget action direction** | "Increase" when CRM notes show qualified leads, positive sentiment, strong conversion signals | "Increase" when notes show "Disqualified", "Not a fit", low projected_value | High — direct spend impact | Practitioner judgment |
| **Semantic reasoning specificity** | Cites exact language from sales notes: "3 of 4 reps noted 'strong budget authority'" | Generic: "positive sentiment detected" or "qualified leads found" | Medium — trust signal | Practitioner judgment |
| **Campaign coverage** | All campaigns from input appear in output with a decision | Any campaign silently absent from output | High — automation blindspot | System requirement |
| **evidence_count accuracy** | Exactly matches session_count from aggregated input data | Any deviation from session_count (higher or lower) | Medium — auditability | Structured output contract |
| **Confidence calibration** | High confidence (0.8–1.0) when all sessions have consistent signal; low (0.2–0.4) when notes are mixed | High confidence on campaigns with 1 session or contradictory notes | Low-Medium — advisory only | Domain intuition |
| **Percentage change magnitude** | Proportional to signal strength (strong unanimous signal → ±40-60%, moderate → ±15-25%) | Flat uniform percentages (all +20%) regardless of signal variance | Low — advisory | Practitioner preference |

### Known Failure Modes in This Domain

1. **Note volume bias** — LLM weights campaigns with more sessions higher even when notes are uniformly negative. High click count + many "Disqualified" notes should be a `pause`, not `increase`.
2. **Generic reasoning on short notes** — When `sales_notes` is brief ("interested", "call back"), the model may produce vague `semantic_reasoning` that doesn't actually cite the note. Looks plausible but is not grounded.
3. **evidence_count fabrication** — Model invents a session count (e.g., "3 sessions reviewed") that doesn't match the actual `session_count` column. Breaks the audit trail.
4. **insufficient_data overuse** — Model retreats to `insufficient_data` / 0% for borderline campaigns to avoid making a call. Unhelpful for automation consumers who need a direction.
5. **Stale context leakage** — In high-throughput API use, if multiple requests share a conversation context (not applicable here — each call is stateless), model outputs for one campaign bleed into another. Not currently a risk but worth monitoring if async batching is added.

### Regulatory / Compliance Context

- **v2.0 scope**: Mock/synthetic data only. No real PII in CRM records for demo. No GDPR/CCPA obligations apply to the current build.
- **Production note**: When real CRM data is ingested (v3+), `sales_notes` may contain prospect PII (names, company details). At that point: data minimization before LLM call, no logging of raw notes, review against GDPR Art. 22 (automated decision-making affecting individuals). Flag for legal review before v3 API consumers send real data.

### Domain Expert Roles for Evaluation

| Role | Responsibility |
|------|---------------|
| Performance marketer (senior, agency) | Dataset labeling — confirm budget action direction correctness; calibrate LLM judge rubric |
| Product owner (Rukesh) | Reference dataset construction from real demo runs; spot-check reasoning quality |
| API consumer (n8n automation author) | Integration test — verify response schema and HTTP semantics meet automation expectations |

## 2. Framework Decision

**Selected Framework:** OpenAI Python SDK (Direct) — `openai>=1.50`

**Version:** `>=1.50,<2.0` (already in `pyproject.toml`)

**Rationale:**
Phase 5 does not introduce new AI architecture. The intelligence layer (`run_analysis()` calling `client.beta.chat.completions.parse()`) was shipped in Phase 3 and is fully functional. Phase 5's AI-relevant work is: (1) exposing `run_analysis()` behind FastAPI endpoints, (2) wiring the two-array input through the existing `pd.merge` + `compute_campaign_agg()` path, and (3) handling Pydantic validation errors at the API boundary. Adding an agent framework on top of a single-turn structured completion call would be pure overhead with no benefit.

The canonical 2025 pattern — `client.beta.chat.completions.parse(response_format=PydanticModel)` — is already the most reliable path for schema-enforced structured output on gpt-4o. No migration is needed.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|-----------------|
| OpenAI Agents SDK | Phase 5 has no agent behavior: no tool calls, no handoffs, no multi-turn reasoning, no streaming. The Agents SDK adds primitives (Agents, Handoffs, Guardrails) that are not needed for a single structured completion. |
| LangChain | Unnecessary abstraction layer over a direct `.parse()` call. Adds 2–3 packages of dependency and 50–100ms latency with zero functionality gain for a single-model call. |
| LangGraph | Stateful graph orchestration is overkill for a synchronous request/response endpoint. Would require redesigning a working, tested function. |
| CrewAI | Multi-agent framework inappropriate for a single structured completion call. |

**Vendor Lock-In Accepted:** Yes — OpenAI gpt-4o is a hard hackathon constraint. Accepted consciously. Mitigation: `AnalysisResult` + `CampaignAction` Pydantic models are provider-neutral; swapping the client call to another provider's SDK would require changing only `_call_llm()` in `llm.py`.

## 3. Framework Quick Reference

> OpenAI Python SDK — Structured Outputs path. Current canonical pattern as of openai>=1.50.

### Installation
```bash
uv add "openai>=1.50,<2.0" "pydantic>=2.8,<3.0"
# Already in pyproject.toml — no new install required for Phase 5
```

### Core Imports
```python
from openai import OpenAI
from pydantic import BaseModel, Field
from typing import Literal
```

### Entry Point Pattern
```python
# EXISTING pattern in llm.py — do not change for Phase 5
client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

completion = client.beta.chat.completions.parse(
    model="gpt-4o-2024-08-06",
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": csv_text},
    ],
    response_format=AnalysisResult,   # Pydantic model — enforced server-side
    max_tokens=2000,
)
result: AnalysisResult = completion.choices[0].message.parsed
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `client.beta.chat.completions.parse()` | Structured Outputs call — guarantees schema-conformant JSON from gpt-4o | Every call to the LLM in this app |
| `response_format=PydanticModel` | Tells the API to enforce the Pydantic v2 schema server-side | Pass the Pydantic class (not an instance) |
| `message.parsed` | The populated Pydantic instance — already validated | Read the result here, not `message.content` |
| `message.refusal` | Non-None when model refuses to answer | Check before reading `.parsed` — raise ValueError if set |
| `tenacity` retry wrapper | Wraps `_call_llm()` to handle 429 / 5xx transiently | Already in place — do NOT add a second retry layer in FastAPI |

### Common Pitfalls

1. **Using `response_format={"type": "json_object"}` (JSON mode) instead of `.parse()`** — JSON mode guarantees valid JSON but NOT schema conformance. The `.parse()` path with a Pydantic class is the right approach and is already in use.
2. **Checking `message.content` instead of `message.parsed`** — In Structured Outputs, `message.content` contains the raw JSON string. `message.parsed` is the validated Pydantic instance. Always use `.parsed`.
3. **Adding retry logic in FastAPI that duplicates `tenacity`** — `_call_llm()` already retries 3 times with exponential backoff. Do not wrap the FastAPI endpoint call in additional retry logic — it will compound wait times on 429 errors.
4. **Passing `gpt-4o` alias instead of a pinned snapshot** — The `gpt-4o` alias may point to a newer model without Structured Outputs support. Always use `gpt-4o-2024-08-06` or newer pinned snapshot.

### Recommended Project Structure
```

├── api/
│   ├── __init__.py
│   ├── main.py          # FastAPI app, endpoint definitions
│   ├── models.py        # API-specific request/response models (imports from llm.py)
│   └── db.py            # psycopg2 connection pool, SQL helpers
```

## 4. Implementation Guidance

**Model Configuration:**
- Model: `gpt-4o-2024-08-06` (pinned snapshot — minimum for Structured Outputs)
- Temperature: default (1.0) — not configurable for Structured Outputs on gpt-4o
- Max tokens: `2000` — sufficient for 5 campaigns with reasoning; increase if > 20 campaigns
- No streaming — synchronous response required for structured output + Pydantic validation

**Core Pattern:**
The FastAPI `POST /api/analyze` endpoint:
1. Receives `{"web_sessions": [...], "crm_records": [...]}` JSON body
2. Converts both arrays to DataFrames with the existing CSV column schema
3. Calls `data.compute_campaign_agg(pd.merge(web_df, crm_df, on="session_id", how="inner", validate="m:1"))`
4. Calls `llm.run_analysis(campaign_agg)` — returns `AnalysisResult` Pydantic instance
5. Returns `result.model_dump()` as the JSON response body

This pattern reuses 100% of the existing data pipeline — the FastAPI endpoint is a thin HTTP wrapper.

**Tool Use:**
- No tool calls. The gpt-4o call is a single-turn structured completion.
- `tenacity` retry is already configured in `llm.py` — inherits automatically.

**State Management:**
- Each request is stateless. No conversation history.
- Results are persisted to Postgres in `analysis_results` table after a successful `run_analysis()` call (see D-10 in CONTEXT.md).
- `pending_sessions` table stores webhook payloads for deferred analysis (D-07/D-11).

**Context Window Strategy:**
- Input: aggregated CSV from `compute_campaign_agg()` — one row per campaign
- Token budget estimate: ~100–150 tokens per campaign row + ~400 tokens system prompt + ~500 tokens buffer = comfortable within 128k context for any realistic campaign count
- If input exceeds 50 campaigns: warn in API response and truncate to top 50 by `session_count DESC` (document this limit in OpenAPI docs)

## 4b. AI Systems Best Practices

> Cross-cutting patterns for the OpenAI SDK + FastAPI integration in Phase 5.

### Structured Outputs with Pydantic

The existing `AnalysisResult` and `CampaignAction` models in `llm.py` are the canonical Pydantic v2 schemas. FastAPI imports them directly — do NOT redefine in `api/models.py`.

```python
# api/models.py — request body only (not the LLM response schema)
from pydantic import BaseModel
from llm import AnalysisResult  # Import, don't redefine

class AnalyzeRequest(BaseModel):
    web_sessions: list[dict]   # Validated downstream by pd.DataFrame constructor
    crm_records: list[dict]

class AnalyzeResponse(BaseModel):
    run_id: str               # UUID for this analysis run
    result: AnalysisResult    # Direct reuse of existing Pydantic model
```

The `.parse()` call enforces `AnalysisResult` server-side — if gpt-4o returns non-conformant JSON, `openai` raises `openai.LengthFinishReasonError` (token limit) or `ValidationError` (schema mismatch). The `tenacity` wrapper retries on `Exception` but not `ValueError` (see `retry_if_not_exception_type(ValueError)` in `llm.py`) — this means schema refusals surface immediately without retry.

### Async-First Design

FastAPI supports async endpoints, but the current `_call_llm()` is synchronous. Options for Phase 5:

- **Sync (current approach, recommended for Phase 5):** Use `def` endpoints in FastAPI with a thread pool. FastAPI automatically runs sync endpoints in a threadpool executor — adequate for low concurrency.
- **Async (future):** Wrap with `asyncio.get_event_loop().run_in_executor()` or migrate to `openai.AsyncOpenAI`. Not needed for Phase 5.

The common mistake: using `async def` with a synchronous blocking call. This blocks the event loop. If `run_analysis()` stays synchronous, use `def` (not `async def`) for the endpoint, or use `run_in_executor()`.

### Prompt Engineering Discipline

- **System prompt**: Defined in `llm.py` `SYSTEM_PROMPT` — do not duplicate or modify in `api/main.py`
- **User message**: The aggregated CSV text from `campaign_agg.to_csv(index=False)` — this is the correct format (CSV is more token-efficient than JSON for tabular data)
- **Prompt versioning**: If the system prompt is modified in v2+, pin a version string in the prompt header so eval comparison is possible: `# Prompt v1.0\nYou are an Autonomous CMO...`
- **Few-shot examples**: Not currently used — the schema enforcement via Structured Outputs is sufficient to get consistent output. Add few-shots only if `semantic_reasoning` quality degrades.

### Context Window Management

- Each API call receives a fresh context — no conversation history accumulation
- The aggregated DataFrame is the entire context: one row per campaign, ~6 columns
- Token-count estimate before call (optional but recommended for large inputs):
```python
import tiktoken
enc = tiktoken.encoding_for_model("gpt-4o")
token_count = len(enc.encode(csv_text)) + len(enc.encode(SYSTEM_PROMPT))
if token_count > 100_000:
    raise HTTPException(422, "Payload too large — reduce campaign count")
```

### Cost and Latency Budget

- **Per-call cost**: gpt-4o at ~$2.50/1M input tokens + ~$10/1M output tokens. A 5-campaign analysis: ~500 input tokens + ~300 output tokens ≈ **$0.004/call**. Negligible for demo scale.
- **Latency**: gpt-4o structured output calls typically 2–6 seconds. Acceptable for synchronous API use. Document in OpenAPI description.
- **Caching strategy**: Postgres `analysis_results` table stores results by `campaign_id + run_id`. A consumer can call `GET /api/campaigns/{campaign_id}/actions` to retrieve the last result without re-calling gpt-4o — the intended read-heavy pattern.
- **Model routing**: Not needed for Phase 5. Future: route to `gpt-4o-mini` for health checks and lightweight validation, reserve `gpt-4o` for full analysis.

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail) | Measurement Approach | Priority |
|-----------|-------------------|---------------------|----------|
| **Schema validity** | `AnalysisResult` Pydantic validates without error; all `budget_action` values in `Literal["increase","pause","decrease","insufficient_data"]`; `confidence` 0.0–1.0; `percentage_change` -100 to +100 | Code — Pydantic `.model_validate()` on every API response | Critical |
| **Campaign coverage completeness** | `len(result.campaigns) == len(input_campaigns)` — all input campaign IDs appear exactly once in output | Code — set comparison of `{c.campaign_id for c in result.campaigns}` vs input | Critical |
| **evidence_count faithfulness** | `c.evidence_count == session_count_from_input[c.campaign_id]` for every campaign | Code — assert in pytest with known fixture | Critical |
| **Budget action direction** | Given a known-signal fixture, `budget_action` matches the expected direction (e.g., all-negative CRM notes → `pause`) | Code (fixture-based) + LLM Judge (on varied inputs) | High |
| **Semantic reasoning specificity** | `semantic_reasoning` contains at least one word or phrase from the input `sales_notes` column for that campaign | LLM Judge: "Does this reasoning cite specific language from the notes? Yes/No" | High |
| **API HTTP semantics** | `POST /api/analyze` with valid key → 200; invalid key → 401; malformed body → 422; `GET /api/health` → 200 with `{"status":"ok"}`; `POST /api/webhook/crm` → 202 | Code — pytest + httpx test client | Critical |
| **Postgres persistence** | After `POST /api/analyze`, a subsequent `GET /api/campaigns/{id}/actions` returns the same result | Code — integration test with real Postgres (via test compose) | High |
| **Latency** | `POST /api/analyze` (live gpt-4o) completes within 15 seconds p95 | Code metric — pytest timing assertion; document in OpenAPI | Medium |

### Reference Dataset

**Size:** 10 examples to start — 5 fixture-based (deterministic, DEMO_MODE), 5 live gpt-4o runs

**Composition:**
- 2 × clean-signal campaigns (all positive notes → `increase`, all negative → `pause`)
- 2 × mixed-signal campaigns (some positive, some negative → expected `decrease` or `insufficient_data`)
- 2 × sparse-data campaigns (1–2 sessions → `insufficient_data` likely)
- 2 × edge cases (empty `sales_notes` → should not hallucinate reasoning)
- 2 × multi-campaign batches (3+ campaigns in one request — tests coverage completeness)

**Labeling:**
- Fixture-based cases (DEMO_MODE): labeled by code assertion against known schema
- Live gpt-4o cases: labeled by product owner review against practitioner rubric above
- LLM judge calibration: run 20 `semantic_reasoning` examples through a GPT-4o judge prompt; validate ≥ 80% agreement with manual review before trusting judge scores

## 6. Guardrails

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **Schema enforcement** | Pydantic `.parse()` fails (schema mismatch, refusal) | `tenacity` retries up to 3x; after 3 failures raises 503 from FastAPI endpoint |
| **API key authentication** | X-API-Key header missing or does not match `API_KEY` env var | Return 401 immediately — no LLM call made |
| **Merge validation** | `pd.merge(validate="m:1")` raises on duplicate session_ids in CRM records | Return 422 with descriptive error: "Duplicate session_id in crm_records" |
| **Empty input guard** | `web_sessions` or `crm_records` is an empty list | Return 422: "web_sessions and crm_records must be non-empty" |
| **Model refusal** | `message.refusal` is non-None | Raises `ValueError` in `_call_llm()` — not retried (by design in tenacity config); surfaces as 500 in FastAPI |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **Budget action distribution** | Log action counts per run to Postgres; review weekly | If `insufficient_data` > 40% of campaigns in a week, investigate prompt or input quality |
| **Semantic reasoning specificity** | Sample 10% of `POST /api/analyze` calls; run LLM judge on `semantic_reasoning` | If judge "not grounded" rate > 20%, add few-shot examples to system prompt |
| **evidence_count mismatch rate** | Code assertion logged per call (should be 0%) | Any non-zero rate → immediate investigation; likely aggregation bug not model issue |
| **API latency trends** | Log gpt-4o call duration from `run_analysis()` | If p95 > 15s sustained, investigate OpenAI service status; consider mini-model fallback |

## 7. Production Monitoring

**Tracing Tool:** Arize Phoenix (open-source, self-hostable) — optional for v2.0 demo scale; structured Postgres logging is sufficient for initial deployment.

For v2.0 (demo/single-tenant): Log each `POST /api/analyze` call to a `api_call_log` table: `id, run_id, campaign_count, latency_ms, budget_actions_json, called_at`. No external tracing tool required.

For v3+ (multi-tenant SaaS): Add Arize Phoenix or Langfuse self-hosted alongside the Postgres service in compose.yaml.

**Key Metrics to Track:**
1. `latency_ms` per `POST /api/analyze` call
2. `budget_action` distribution per week (increase/pause/decrease/insufficient_data counts)
3. `429_rate` — OpenAI rate limit errors caught by tenacity (signals API key quota pressure)
4. `schema_validation_failure_rate` — Pydantic parse errors after tenacity exhaustion (signals model degradation)
5. `webhook_queue_depth` — count of unprocessed rows in `pending_sessions` (signals webhook consumers not calling analyze)

**Alert Thresholds:**
- `latency_ms > 15000` sustained for 5 consecutive calls → check OpenAI status page
- `429_rate > 10%` in a rolling 1-hour window → upgrade OpenAI API tier
- `schema_validation_failure_rate > 0%` in any 24-hour window → immediate investigation
- `webhook_queue_depth > 100` → notify owner — pending sessions not being processed

**Smart Sampling Strategy:**
- Always sample calls where `schema_validation_failure_rate > 0`
- Sample 100% of calls during the first 48 hours after deployment
- After stabilization: sample 10% randomly + 100% of calls with `latency_ms > 10000`
- Human review focus: calls where `insufficient_data` rate is unexpectedly high
