<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Untrusted browser/HTTP client → public POST /api/waitlist | No API key required; anyone on the internet can POST |
| Untrusted Origin header → CORSMiddleware | Wildcard origin policy — any cross-origin frontend can call the endpoint |
| FastAPI handler → api.db.insert_waitlist_email | Email value (already EmailStr-validated) crosses into SQL via parameterized binding |
| FastAPI handler → api.email_utils.send_waitlist_notification | Email + DB-generated timestamp cross into SMTP envelope |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-09 | Spoofing | POST /api/waitlist accepts any Origin / no auth | accept | D-12 explicitly makes this endpoint public — landing page visitors don't have API keys. WAIT-01 requires immediate confirmation. CORS wildcard (D-11) accepted for credential-free MVP per CONTEXT.md tradeoff. Future hardening (ALLOWED_ORIGINS env var, hCaptcha) is deferred per CONTEXT §Deferred. Single-tenant demo scope |
| T-06-10 | Tampering | Email-header injection via `\r\n` in email field reaching SMTP envelope construction | mitigate | `EmailStr` (Plan 01 Task 3 + this plan Task 2 test_waitlist_invalid_email_returns_422) rejects addresses containing CR/LF before the handler body runs. The handler interpolates `body.email` into `Subject:` and body text in api/email_utils.py, but the value has already been validated as RFC-compliant by the model layer |
| T-06-11 | Information Disclosure | SMTP failure detail leaks internal info via 500 response | accept | D-06 — "fail loudly" is the explicit choice. The detail `f"SMTP error: {exc}"` may reveal SMTP host or auth state but exposure is limited to operators triggering misconfigured deployments. Single-tenant MVP, no PII in payload. Multi-tenant hardening deferred |
| T-06-12 | Denial of Service | Unauthenticated flood of POST /api/waitlist signups (rate-limit absent) | accept | UNIQUE constraint blocks row growth from a single attacker's email. Distinct-email flooding is possible; rate limiting is explicitly deferred per CONTEXT.md §Deferred ("Rate limiting on /api/waitlist — Prevent spam signups. Single-tenant demo doesn't need it yet."). Caddy edge rate limiting can be added in Phase 8 if needed |
| T-06-13 | Tampering | Test harness patching wrong module binding masks real bug | mitigate | Pitfall 5 — tests use `db_overrides` dict applied via `monkeypatch.setattr(f"api.main.{name}", mock)`. Task 2 acceptance criterion explicitly requires `_make_client(monkeypatch, db_overrides={"insert_waitlist_email": ...})` and forbids patching `api.db.insert_waitlist_email` directly. Aligns with existing Phase 5 pattern |
| T-06-14 | Repudiation | Order of operations: SMTP send before DB commit could send email but lose the row | mitigate | Handler structure: `signed_up_at = insert_waitlist_email(body.email)` returns AFTER `with conn:` exits cleanly (commit point in psycopg2). Only then does the handler call `send_waitlist_notification(...)`. Anti-pattern called out explicitly in RESEARCH §Anti-Patterns is averted by the sequential call structure mandated in Task 1 |
| T-06-15 | Tampering | Future plans accidentally remove the public-endpoint property of /api/waitlist by adding Depends(verify_api_key) | mitigate | Acceptance criterion in Task 1: handler signature has exactly one parameter named `body`. Acceptance criterion in Task 2: `test_waitlist_valid_email_returns_200` posts without X-API-Key header and expects 200 — any future regression that adds auth dependency will turn this test red |
