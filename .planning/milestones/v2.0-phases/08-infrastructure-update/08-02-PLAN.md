<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Host network → containers | Only the `caddy` service publishes ports (80/443); all other services are internal-bridge-only. |
| Compose env file → containers | `${POSTGRES_PASSWORD}` substitution reads from the on-host `.env` file (per docker-compose default behavior). The `.env` file must not be present in any container's build context (handled by root `.dockerignore` and Plan 01's `landing/.dockerignore`). |
| Service-to-service (internal bridge) | All inter-service traffic (caddy↔app, caddy↔fastapi, caddy↔landing, fastapi↔postgres) crosses the Docker internal bridge network. No TLS between containers — acceptable because the bridge is host-local and not exposed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-06 | Elevation of Privilege | landing service host port exposure | mitigate | Acceptance criterion enforces `services.landing` does NOT contain a `ports:` key. If someone adds `ports: ["3000:3000"]` they bypass Caddy's TLS termination, expose the unsecured Next.js server directly to the internet, and circumvent the route-dispatch trust boundary. Mechanical check via YAML parser asserts absence. (D-03, RESEARCH.md Anti-Patterns.) |
| T-08-07 | Information Disclosure | landing env_file | mitigate | Acceptance criterion enforces `services.landing` does NOT contain an `env_file:` key. The Next.js standalone server reads no server-side secrets — NEXT_PUBLIC_* values must NOT be set at runtime (they bake at build time, and D-14 from Phase 7 mandates they remain unset). Adding `env_file: .env` would load `OPENAI_API_KEY`, `POSTGRES_PASSWORD`, and SMTP creds into the landing container's process env where any client-side rendering bug could leak them. |
| T-08-08 | Denial of Service | fastapi startup race against postgres cold-start | mitigate | `services.fastapi.depends_on.postgres.condition: service_healthy` (dict form per Pitfall 6) gates fastapi startup on `pg_isready` success. Without this, FastAPI's `init_db()` lifespan handler attempts to CREATE TABLE before postgres accepts connections and the container crashes — `restart: unless-stopped` then loops on restart-backoff. Acceptance criterion enforces dict form AND `service_healthy` literal. |
| T-08-09 | Tampering | postgres healthcheck command | mitigate | `pg_isready -U ppuser -d performance_plus` matches the POSTGRES_USER/POSTGRES_DB env vars on the same service. If a future change renames the user/db without updating the healthcheck, pg_isready returns a different exit code and the service never reports healthy — making the change loud rather than silent. Acceptance criterion compares the exact test command via YAML parser. |
| T-08-10 | Repudiation | caddy depends_on form silent drop | accept | If a future edit changes caddy depends_on from list form to dict form WITHOUT adding a `condition:` key, Compose may parse but the ordering semantics are unchanged. D-10 documents the intent (list form, no health gating for caddy). Risk is low (no functional impact); test coverage in the acceptance criterion catches the inverse mistake (list vs dict). |

**No HIGH threats remain unmitigated.** T-08-06 through T-08-09 are enforced by mechanical acceptance_criteria checks parsing compose.yaml as YAML.
</threat_model>
