<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser visitor email input → WaitlistForm state → fetch body → FastAPI EmailStr | Untrusted user input crosses this boundary. Multiple validation layers (browser `type="email"`, Pydantic `EmailStr` server-side) prevent malformed payloads. |
| Same-origin (production) / cross-origin (local dev) fetch → FastAPI POST | Local dev: `localhost:3000` → `localhost:8000`. Production: same-origin `/api/waitlist` via Caddy. |
| External link `<a href="/app" target="_blank">` → Streamlit | Browser tab opens a new window onto the same origin (production) or a broken link (local dev). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-09 | Tampering (XSS) via React string children | All hardcoded strings in HeroSection, HowItWorksSection, WaitlistForm | accept | React auto-escapes string children rendered as JSX text. No `dangerouslySetInnerHTML` used anywhere in this plan (verified by acceptance criterion). |
| T-07-10 | Input Validation — malformed email submitted to /api/waitlist | WaitlistForm fetch payload `{ email }` | mitigate | Two-layer validation: (1) browser `<input type="email" required>` rejects malformed before submit, (2) FastAPI `WaitlistRequest` model uses Pydantic `EmailStr` (Phase 6, verified in PATTERNS.md) which returns 422 on invalid input. WaitlistForm treats any non-200/409 response as `"error"` state, surfacing the generic message — never echoes raw `detail` from the server (D-12 forbids exposing API error detail). |
| T-07-11 | Information Disclosure — API error detail leaked to visitor | Error branch in WaitlistForm | mitigate | The form maps `res.ok` → `"success"`, `res.status === 409` → `"duplicate"`, everything else → `"error"` with a hardcoded generic message. The response body is never parsed or shown — `res.json()` is not called. Even SMTP failure messages (`"SMTP error: ..."` in api/main.py) remain server-side. |
| T-07-12 | Tab-nabbing via `target="_blank"` on Try Demo CTA | `<a href="/app" target="_blank">` | mitigate | `rel="noopener noreferrer"` is attached to the Try Demo anchor — prevents `window.opener` access from the newly opened tab. Enforced via acceptance criterion. |
| T-07-13 | CSRF on waitlist endpoint via cross-origin POST | POST /api/waitlist | accept | Endpoint is public, idempotent on the user's intent (creating a waitlist signup), and returns 409 on duplicate (no destructive side effect). CORS wildcard at the API is acceptable for v2.0 per Phase 6 D-05 and PATTERNS.md §S-5. No CSRF token required for a hackathon-stage public waitlist; revisit when launching auth in a future phase. |
| T-07-14 | Hydration mismatch in WaitlistForm | useState initial value | mitigate | Initial state is the static literal `"idle"` (no `window` access, no `process.env.NEXT_PUBLIC_*` access during render that varies between server and client). `process.env.NEXT_PUBLIC_*` reads occur inside `handleSubmit`, not during render. Matches RESEARCH.md §Pitfall 5 guidance. |
</threat_model>
