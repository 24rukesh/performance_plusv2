## 1. System Classification

**System Type:** Conversational (existing single-model structured-output call with auto-fallback resilience)

**Description:**
Performance Plus uses gpt-4o via `client.beta.chat.completions.parse()` to generate structured budget action recommendations from merged ad analytics + CRM data. Phase 12 adds runtime resilience to this call: any `openai.OpenAIError` (rate limit, auth, connection failure) silently falls back to a pre-validated fixture response, keeping the demo fully functional offline. Users are the marketers running the Streamlit app who need reliable analysis output regardless of API availability.

**Critical Failure Modes:**
1. Fallback silently activates even when the API is reachable — user never sees live recommendations
2. Fixture response fails `AnalysisResult.model_validate()` after fallback, crashing the results section instead of recovering gracefully
3. Session state keys (`analysis_result`, `campaign_agg`, `merged_df`) are not restored correctly after Postgres load, so Charts / Filters / Export tabs break on reload

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Performance marketing / B2B demand generation

**User Population:** B2B performance marketers and CMOs who manage paid ad budgets across Google Ads, Meta Ads, and LinkedIn. They use the app during sales conversations, internal strategy reviews, and post-campaign debriefs to justify budget routing decisions to finance and executive stakeholders.

**Stakes Level:** High

**Output Consequence:** The marketer acts on AI-generated budget action recommendations (increase / pause / decrease by %) across active ad campaigns. An incorrect recommendation routes real ad spend to under-performing campaigns or pauses campaigns generating qualified pipeline. A reloaded analysis that silently breaks interactive features (charts, exports) means the marketer cannot defend the recommendation to a CFO or agency client. A fallback fixture displayed without clear labeling means the marketer may act on stale or synthetic data as if it were a live analysis.

### What Domain Experts Evaluate Against

**Dimension: Recommendation traceability to sales evidence**
Good (domain expert would accept): Each budget action names the specific sales-note language or CRM signal (e.g., "3 of 4 leads from Campaign X described 'budget approved Q3'") that drove the recommendation. The `semantic_reasoning` field reads as a one-sentence brief a sales rep could verify against their own notes.
Bad (domain expert would flag): Recommendation says "increase budget" with reasoning like "campaign shows positive signals" — no link to the CRM data the marketer uploaded. Expert cannot tell if the AI used the sales notes or ignored them.
Stakes: Critical
Source: B2B marketing attribution practice — 40% of CMOs now require demonstrable attribution across every element of the marketing mix (eMarketer 2025 via Typeface.ai CMO ROI guide). Unsourced AI budget recommendations carry the same credibility risk as gut-feel decisions, which the tool is explicitly replacing.

**Dimension: Fallback transparency — live vs. cached result**
Good (domain expert would accept): When the OpenAI API is unavailable, a visible, persistent banner states "OpenAI API unavailable — showing cached demo results" above all recommendation output. The marketer knows before acting that numbers are from a fixture, not their uploaded data.
Bad (domain expert would flag): Fixture results are displayed with the same visual treatment as live results. A marketer in a sales call presents "their" campaign recommendations; the prospect asks about Campaign B, which doesn't exist in the fixture. The discrepancy surfaces mid-conversation.
Stakes: Critical
Source: Stanford HAI governance guidance — AI systems must make their operating mode (live inference vs. cached/demo) transparent to users before they act on outputs. Governance failure at this point erodes trust that cannot easily be rebuilt in a sales context.

**Dimension: Saved analysis fidelity on reload**
Good (domain expert would accept): Loading a saved analysis from the sidebar restores the full interactive state: charts render with correct data, platform filters work, comparison and drill-down features work, PDF and CSV exports produce correct output. The reloaded state is indistinguishable from a freshly-run analysis.
Bad (domain expert would flag): Charts tab shows blank or mis-labeled axes after reload. Filters show options from the current session's data rather than the saved run's data. PDF export produces an empty or truncated report. Expert discovers the discrepancy when preparing a client presentation from a saved run.
Stakes: High
Source: Domain practice — marketers save analyses specifically to reuse them in agency/client reporting and finance reviews. A reload that silently degrades interactive features makes the persistence feature worse than no persistence at all.

**Dimension: Budget action decisiveness — avoidance of non-answers**
Good (domain expert would accept): Every campaign with at least one CRM-attributed session receives a directional recommendation (increase, pause, or decrease) with a non-zero percentage change and a confidence score above 0.5.
Bad (domain expert would flag): Multiple campaigns return `insufficient_data` with `percentage_change: 0` even though CRM notes are present for those campaigns. Marketer receives a tool output that requires them to manually re-examine the same data the tool was given — no time saved.
Stakes: High
Source: Writer.com analysis of AI failure modes in marketing — "AI tools that return hedged or non-committal outputs on well-evidenced inputs undermine user trust faster than confident-but-wrong outputs, because they offer no decision value." Practitioner expectation is AI as decision support, not AI as a mirror of their own uncertainty.

**Dimension: Session persistence correctness — no silent data loss**
Good (domain expert would accept): A saved analysis reloaded 24 hours later returns exactly the same `campaign_id` list, `budget_action` values, `percentage_change` values, and `semantic_reasoning` strings as when it was saved. The Postgres round-trip is byte-for-byte lossless for all three payload objects.
Bad (domain expert would flag): A reloaded analysis shows different `percentage_change` values than what the marketer remembered approving. Floating-point serialization or DataFrame column type drift during JSONB round-trip has silently altered numbers. Marketer presents a recommendation that does not match their approved version.
Stakes: High
Source: B2B marketing attribution standard — enterprise attribution tools are evaluated on audit trail integrity (Improvado B2B Attribution Guide 2026). Any tool that modifies data during a save/load cycle fails basic auditability requirements.

### Known Failure Modes in This Domain

1. **Silent fallback without user notification.** The API fallback fires, fixture data is displayed, and the marketer runs a strategy session or client call based on synthetic data — not their uploaded campaigns. This is the single most trust-damaging failure mode for a sales demo tool, because the moment of discovery (a campaign the prospect asks about that doesn't exist in the fixture) happens in front of an audience.

2. **Recommendation drift under fixture reuse.** When the same fixture is shown across multiple demo sessions with different prospects, domain experts notice that "their" campaign names and percentages match what was shown in a previous demo. The fixture must use plausible but clearly labelled synthetic campaign IDs, and the banner must be persistent enough that no user mistakes fixture output for live output.

3. **DataFrame column type corruption on Postgres round-trip.** Numeric columns (spend, `percentage_change`, `confidence`) serialized to JSONB as Python floats may deserialize as strings or lose precision if `orient="records"` is used without explicit dtype handling. The restored DataFrame passes visual inspection but fails downstream arithmetic in chart rendering or export calculations.

4. **Stale saved analysis used for a live budget decision.** A marketer loads an analysis saved six weeks ago (when campaign mix was different), does not notice the `saved_at` timestamp in the sidebar, and uses the recommendations for the current month's budget. The system cannot prevent this, but the sidebar display must make `saved_at` visible on every entry — not buried in a tooltip — so the marketer has the signal they need.

### Regulatory / Compliance Context

No sector-specific regulation (HIPAA, FCA, MiFID II) applies to this deployment context. The system processes synthetic/mock data and produces marketing budget recommendations, not financial advice or medical information.

Directly relevant governance standard: **GDPR Article 22** (automated decision-making) is not triggered here because budget routing decisions are reviewed and enacted by a human marketer — the AI output is advisory, not automated. If the system were to auto-submit budget changes via a platform API (Meta Ads API, Google Ads API), Article 22 and sector ad-spend regulations would apply. That path is explicitly deferred to v4+ per REQUIREMENTS.md.

Operational governance requirement (non-regulatory but critical for enterprise sales): the app must make fallback vs. live mode distinguishable to all viewers present during a demo, not just the operator. This is a sales trust requirement, not a legal one.

## 2. Framework Decision

**Selected Framework:** OpenAI Python SDK (openai>=1.50,<2.0 — already installed)

**Version:** >=1.50,<2.0 (pinned in pyproject.toml)

**Rationale:**
Phase 12's AI scope is a single `try/except openai.OpenAIError` block around the existing `client.beta.chat.completions.parse()` call in `llm.py`. The OpenAI SDK already provides the complete exception hierarchy (`AuthenticationError`, `RateLimitError`, `APIConnectionError`, `APIStatusError`) needed for D-07 and D-09. No orchestration framework (LangChain, LangGraph, CrewAI, etc.) adds any value here — they are abstractions over the direct SDK call that is already correctly implemented. Adding one would increase dependency surface, introduce abstraction overhead, and violate the CLAUDE.md constraint of not adding infrastructure beyond what is needed.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| OpenAI Agents SDK | Orchestration abstraction for multi-agent workflows — Phase 12 has one API call with error handling, not agent delegation |
| LangChain | General-purpose LLM pipeline framework — adds abstraction overhead with no benefit for a single try/except wrapper |
| LangGraph | Stateful graph workflows — overkill for a linear call with fallback |
| tenacity (retry library) | Already installed; Phase 12 uses it for retries in other contexts, but the fallback-to-fixture is intentional not a retry |

**Vendor Lock-In Accepted:** Yes — OpenAI gpt-4o is a fixed hackathon constraint (CLAUDE.md). Lock-in is a project requirement, not a trade-off.

### Entry Point Pattern

The Phase 12 entry point is wrapping the existing `_call_llm()` call in `run_analysis()` with a `try/except openai.OpenAIError` block and falling back to `_load_fixture()` on any failure. This is the canonical pattern from the OpenAI Python SDK README:

```python
import openai
import logging
import st  # Streamlit — only available inside a Streamlit run context

logger = logging.getLogger(__name__)


def run_analysis(campaign_agg: pd.DataFrame) -> AnalysisResult:
    """Run gpt-4o analysis; fall back to fixture on any OpenAI error (D-07)."""
    # Existing explicit DEMO_MODE path — unchanged
    if os.environ.get("DEMO_MODE") == "1" and client is None:
        return _load_fixture()

    t0 = time.perf_counter()
    csv_text = campaign_agg.to_csv(index=False)

    try:
        result = _call_llm(csv_text)          # tenacity retries 3x inside here
    except openai.OpenAIError as exc:          # D-09: base class catches all subclasses
        logger.warning(
            "OpenAI API unavailable — falling back to fixture. "
            "Error type: %s | message: %s",
            type(exc).__name__,
            str(exc),
        )
        # Set session state flag so app.py can render st.warning (D-08)
        import streamlit as st
        st.session_state["api_fallback_active"] = True
        return _load_fixture()                 # always passes model_validate()

    logger.info(
        "run_analysis completed in %.2fs | campaigns=%d",
        time.perf_counter() - t0,
        len(result.campaigns),
    )
    return result
```

Note: `import streamlit as st` is placed inside the except branch intentionally. `llm.py` is imported at module level by `app.py`; a top-level `import streamlit` in `llm.py` would create a circular dependency risk and break any headless/test usage of `llm.py`.

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `openai.OpenAIError` | Base class for every exception the SDK raises. Subclasses cover auth (401), rate limit (429), connection failure, timeout, and all other non-2xx responses. | The single `except` clause in `run_analysis()` — catches all current and future SDK errors in one line (D-09). |
| `openai.APIStatusError` | Subclass of `OpenAIError` for any non-2xx HTTP response that is not already mapped to `AuthenticationError` or `RateLimitError`. Has `.status_code` and `.response` properties. | Inspect `exc.status_code` in logging if you need to distinguish a 503 (server down) from a 400 (bad request) in future diagnostics. |
| `openai.APIConnectionError` | Raised when the SDK cannot reach the API at all — DNS failure, TCP timeout, SSL error. Wraps `exc.__cause__` from the underlying `httpx` transport. | Network-level fallback trigger. `exc.__cause__` gives the raw httpx error for deeper diagnosis. |
| `client.beta.chat.completions.parse()` | The Structured Outputs entry point. Converts the `response_format=AnalysisResult` Pydantic class into a `json_schema` with `strict: True` server-side, then deserialises the response back into a typed `AnalysisResult` instance. | Every live gpt-4o call in `_call_llm()`. Never use `.create()` with `response_format={"type":"json_object"}` — that only guarantees valid JSON, not your schema. |
| `AnalysisResult.model_validate(data)` | Pydantic v2 class method. Accepts a `dict` (e.g. from `json.loads()`) and returns a fully-validated `AnalysisResult` instance, raising `ValidationError` on schema mismatch. | `_load_fixture()` uses this so the fixture return value is type-identical to the live API path. Any schema change must also update the fixture file. |

### Common Pitfalls

1. **Catching `Exception` instead of `openai.OpenAIError`** — `except Exception` silently swallows `KeyboardInterrupt`, `SystemExit`, Pydantic `ValidationError` from a broken fixture, and every other non-OpenAI error. The fallback fires for unrelated bugs and masks the root cause. Always catch the SDK's own base class: `except openai.OpenAIError`.

2. **Forgetting `api_fallback_active` reset on new analysis run** — If the user uploads new CSVs and triggers a fresh analysis after a previous fallback, `api_fallback_active` stays `True` from the prior run. The `st.warning` banner then appears on a successful live response. Reset `st.session_state["api_fallback_active"] = False` at the top of the `try` block (before `_call_llm()`), or in the `app.py` "Run Analysis" button handler before calling `run_analysis()`.

3. **Fixture failing `model_validate()` silently after a schema change** — When you add a new required field to `CampaignAction` or `AnalysisResult` (e.g. `source_platforms` was added in Phase 10), the fixture JSON must be updated too. If it is not, `_load_fixture()` raises `ValidationError` inside the `except` branch, crashing the results section instead of recovering. Test `_load_fixture()` in `tests/test_llm.py` on every schema change.

4. **`tenacity` retrying `openai.OpenAIError` that should immediately fall back** — The `@retry` decorator on `_call_llm()` already uses `retry_if_not_exception_type(ValueError)`, meaning it will retry on `OpenAIError`. This is correct for transient 429/503. But the `try/except` in `run_analysis()` wraps the entire `_call_llm()` call including all tenacity retries — so the fallback only fires after all 3 retry attempts are exhausted. This is the intended behavior. Do not move the `try/except` inside `_call_llm()`.

5. **Setting `st.session_state` from inside `llm.py` without a Streamlit context** — `st.session_state` only exists during a live Streamlit app run. If `run_analysis()` is called from a pytest test, accessing `st.session_state` raises `StreamlitAPIException`. Guard the session state write with `hasattr(st, "session_state")` or mock `st` in tests. The import inside the `except` branch (not at module top) already avoids the import-time crash.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### 4b.1 Structured Outputs with Pydantic

The project uses OpenAI's Structured Outputs path via `client.beta.chat.completions.parse()` with a Pydantic v2 `BaseModel` as `response_format`. This is the correct 2025 pattern for `gpt-4o-2024-08-06` and newer snapshots.

**The schema (already in `llm.py`):**

```python
from typing import Literal
from pydantic import BaseModel, Field

class CampaignAction(BaseModel):
    campaign_id: str = Field(description="Matches campaign_id from input CSV exactly")
    budget_action: Literal["increase", "pause", "decrease", "insufficient_data"]
    percentage_change: int = Field(
        description="Signed integer -100 to +100. pause=-100, insufficient_data=0."
    )
    semantic_reasoning: str = Field(
        description="One sentence citing specific language from sales notes."
    )
    confidence: float = Field(ge=0.0, le=1.0)
    evidence_count: int
    source_platforms: list[str]

class AnalysisResult(BaseModel):
    executive_summary: str
    campaigns: list[CampaignAction]
```

**How the SDK uses it:**

Passing `response_format=AnalysisResult` to `.parse()` causes the SDK to:
1. Convert the Pydantic class to a `json_schema` with `strict: True` server-side.
2. Return `completion.choices[0].message.parsed` as a fully-validated `AnalysisResult` instance — no `json.loads()` needed.
3. Set `message.refusal` (non-None string) instead of `message.parsed` if the model refuses to comply.

**Refusal check (already in `_call_llm()`):**

```python
message = completion.choices[0].message
if message.refusal:
    # Raise ValueError so tenacity does NOT retry (retry_if_not_exception_type(ValueError))
    raise ValueError(f"Model refused to analyse: {message.refusal}")
return message.parsed  # type: AnalysisResult, guaranteed by SDK
```

**Fixture round-trip validation (already in `_load_fixture()`):**

```python
def _load_fixture() -> AnalysisResult:
    fixture_path = Path(__file__).parent / "data" / "fixture_results.json"
    data = json.loads(fixture_path.read_text())
    # model_validate() raises pydantic.ValidationError if schema is out of sync
    return AnalysisResult.model_validate(data)
```

**Retry logic for validation failure:** The `.parse()` method enforces the schema server-side with `strict: True`, so `ValidationError` from `.parse()` itself should not occur in practice. If it does (e.g. an SDK bug), `tenacity` will retry it because `retry_if_not_exception_type(ValueError)` only excludes `ValueError` — `ValidationError` is not `ValueError`. After 3 failed tenacity retries, the `openai.OpenAIError`-or-`ValidationError` exception propagates to `run_analysis()`'s `except openai.OpenAIError` block. If it is a `ValidationError` (not an `OpenAIError`), it will not be caught there and will surface as an unhandled exception. Add an explicit `except pydantic.ValidationError` branch if schema-level failures need fallback treatment.

**Phase 12 specific:** The fixture file at `data/fixture_results.json` must pass `AnalysisResult.model_validate()` after every schema change. Add a test:

```python
# tests/test_llm.py
def test_fixture_validates():
    result = _load_fixture()
    assert isinstance(result, AnalysisResult)
    assert len(result.campaigns) > 0
```

### 4b.2 Async-First Design

**Streamlit is synchronous — this is a hard constraint for Phase 12.**

Streamlit's execution model re-runs the entire `app.py` script from top to bottom on every user interaction. It does not run inside an `asyncio` event loop. This means:

- `client.beta.chat.completions.parse()` (the synchronous OpenAI client) is the correct choice. Do not use `AsyncOpenAI` here.
- Do NOT call `asyncio.run(async_func())` inside a Streamlit callback. Streamlit may already be running an event loop internally (depending on version), and `asyncio.run()` inside an existing loop raises `RuntimeError: This event loop is already running`.
- `tenacity`'s `@retry` decorator works correctly with synchronous functions and does not require any async changes.

The one common mistake: developers copy-paste async examples from the OpenAI SDK README (`await client.chat.completions.create(...)`) into Streamlit callbacks. This will fail with `RuntimeError` or silently hang. Use the synchronous `OpenAI` client, not `AsyncOpenAI`.

**Stream vs. await:** For this use case, streaming is not used. `client.beta.chat.completions.parse()` does not support streaming (`stream=True` and `response_format=PydanticModel` are mutually exclusive — the parsed Pydantic instance is only available after the full response is received). Streaming is appropriate for conversational UX (token-by-token display), not for structured output validation.

### 4b.3 Prompt Engineering Discipline

**No changes to the prompt are made in Phase 12.** The system prompt and user message template in `llm.py` are locked from Phase 10. Document the existing discipline for reference:

**System vs. user prompt separation:** The system prompt (`SYSTEM_PROMPT`) contains the persona definition, decision rules, and output format constraints. The user message contains only the data payload (CSV text). This separation ensures:
- The model cannot be confused about whether rules are "data" or "instructions."
- Prompt updates (e.g. changing budget decision rules) are localized to `SYSTEM_PROMPT` and do not require touching the data serialization logic.

**Few-shot examples:** Not used for this system. The Structured Outputs schema + explicit rules in `SYSTEM_PROMPT` are sufficient to constrain the output format. Few-shot would add tokens and increase cost without a demonstrated quality improvement for this use case.

**`max_tokens` is explicitly set to `2000`.** Never leave `max_tokens` unbounded in production. A gpt-4o call without `max_tokens` can consume the full model output budget (~4096 tokens for some snapshots, ~16384 for others), increasing cost by 2-8x on verbose responses. For this app, each campaign requires ~100-150 output tokens; 2000 is safe for up to ~15 campaigns.

### 4b.4 Context Window Management

The tiktoken preflight guard is already implemented in Phase 10 (`count_prompt_tokens()` in `llm.py`). Phase 12 does not change the context window strategy.

**Current approach:** Token count is computed before the API call using `o200k_base` encoding (correct for `gpt-4o-2024-08-06`). The encoding mirrors the exact message format sent to the model:

```python
def count_prompt_tokens(campaign_agg: pd.DataFrame) -> int:
    enc = tiktoken.get_encoding("o200k_base")
    csv_text = campaign_agg.to_csv(index=False)
    user_msg = f"Analyse these campaigns and return your budget decisions:\n\n{csv_text}"
    return len(enc.encode(SYSTEM_PROMPT)) + len(enc.encode(user_msg))
```

**For this system type (single-call structured output):** Context window pressure comes entirely from the input payload (number of campaigns x columns in `campaign_agg`). The output is bounded by `max_tokens=2000`. Primary mitigation is the pre-aggregation step that reduces the input to one row per `campaign_id`.

If token count approaches the 128k window (unlikely for normal datasets but possible with many campaigns and many source platforms), options in priority order:
1. Drop low-signal columns from `campaign_agg` before serialization.
2. Pre-summarize `sales_notes` per campaign to reduce text length.
3. Split campaigns into batches of N and make multiple API calls (adds latency and cost).

### 4b.5 Cost and Latency Budget

**Per-call cost estimate (gpt-4o-2024-08-06, as of 2025):**

| Cost component | Estimate |
|---------------|---------|
| Input tokens | ~2,000–5,000 tokens (system prompt ~700 + CSV payload ~1,300–4,300 depending on campaign count) |
| Output tokens | ~500–2,000 tokens (one `CampaignAction` per campaign, ~100–150 tokens each) |
| Input token price | $2.50 / 1M tokens |
| Output token price | $10.00 / 1M tokens |
| **Cost per analysis run** | **~$0.01–$0.03** |

At 100 analysis runs/day: ~$1–3/day. At 1,000 runs/day (SaaS scale): ~$10–30/day.

**Fallback has zero cost.** When `api_fallback_active` is True, `_load_fixture()` reads a local JSON file — no API call, no cost, no latency (sub-millisecond).

**Caching strategy:**

For the hackathon MVP, no response caching is implemented. Caching is appropriate for SaaS v2 when:
- The same `campaign_agg` CSV content is submitted multiple times (e.g. user re-runs without changing data).
- Cache key: SHA-256 hash of the CSV text + model name.
- Cache store: Redis (v2 SaaS stack per CLAUDE.md) or Postgres JSONB with a hash index.
- TTL: 1 hour (campaign data changes daily at most).

**Sub-task model routing (SaaS v2 pattern, not Phase 12):** gpt-4o-mini supports Structured Outputs and costs ~10x less than gpt-4o. For a "preview mode" (CLAUDE.md SaaS v2 pattern), run gpt-4o-mini first for a fast first pass, then escalate to gpt-4o for final budget decisions. Classification and routing tasks (e.g. "does this campaign have enough data?") should always use the cheaper model.

## 5. Evaluation Strategy

### Dimensions

The following dimensions are grounded in the Section 1b domain rubric ingredients. Each dimension maps to a specific failure mode identified by the domain researcher. Dimensions are ordered by consequence: Critical failures are demo-breaking; High failures degrade auditability and reusability.

| Dimension | Source (Section 1b) | Measurement | Priority |
|-----------|---------------------|-------------|----------|
| Fallback trigger selectivity | Critical failure mode 1 — silent fallback | Code | Critical |
| Fallback banner visibility | Fallback transparency rubric | Code | Critical |
| Fixture schema compliance | Critical failure mode 2 — fixture crashes | Code | Critical |
| Session state restoration fidelity | Critical failure mode 3 — reload breaks tabs | Code | High |
| Postgres round-trip integrity | Session persistence correctness rubric | Code | High |
| Recommendation traceability | Recommendation traceability rubric | LLM Judge | High |
| Budget action decisiveness | Budget action decisiveness rubric | Code | High |

#### Dimension 1: Fallback Trigger Selectivity

**What it measures:** The `except openai.OpenAIError` clause catches only genuine OpenAI SDK exceptions — not unrelated Python exceptions like `KeyboardInterrupt`, `pydantic.ValidationError`, or `ValueError` from a broken fixture. Catching too broadly masks real bugs with a silently successful fallback.

**PASS:** When `run_analysis()` is called and a mock raises `openai.RateLimitError`, `openai.AuthenticationError`, `openai.APIConnectionError`, `openai.APIStatusError`, or `openai.APITimeoutError`, the function returns `_load_fixture()` output and sets `api_fallback_active=True`. When a mock raises `ValueError`, `pydantic.ValidationError`, `KeyboardInterrupt`, or `RuntimeError`, those exceptions propagate unhandled (do not trigger fallback).

**FAIL:** A non-OpenAI exception (e.g., `pydantic.ValidationError` from a broken fixture) causes `api_fallback_active=True` to be set and `_load_fixture()` to be returned, masking the real error and preventing the developer from knowing the fixture is broken.

**Measurement:** Code — pytest with `unittest.mock.patch` to raise each exception subclass. Assert return type and `api_fallback_active` state for each case.

```python
# tests/test_evals.py
import pytest
from unittest.mock import patch, MagicMock
import openai
from llm import run_analysis

@pytest.mark.parametrize("exc_class", [
    openai.RateLimitError,
    openai.AuthenticationError,
    openai.APIConnectionError,
])
def test_openai_errors_trigger_fallback(exc_class, mock_st_session_state):
    with patch("llm._call_llm", side_effect=exc_class("mock error", response=MagicMock(), body={})):
        result = run_analysis(mock_campaign_agg)
    assert mock_st_session_state.get("api_fallback_active") is True

def test_value_error_does_not_trigger_fallback(mock_campaign_agg):
    with patch("llm._call_llm", side_effect=ValueError("bad input")):
        with pytest.raises(ValueError):
            run_analysis(mock_campaign_agg)
```

#### Dimension 2: Fallback Banner Visibility

**What it measures:** When `api_fallback_active=True` is set in session state, `app.py` renders `st.warning(...)` above the tabs before any campaign recommendation is visible. No banner should appear when `api_fallback_active=False`.

**PASS:** After a simulated API failure, the Streamlit UI shows a persistent `st.warning` element containing the string "OpenAI API unavailable" positioned above the `st.tabs()` call. The banner text is distinguishable from the existing `st.info` banner used for the explicit `DEMO_MODE=1` env var path. When `api_fallback_active=False` (live API succeeded), no fallback banner renders.

**FAIL:** Fixture results render with identical visual treatment to live results — no banner, or banner appears only on hover/tooltip. A marketer in a sales call with three prospects visible on screen cannot distinguish cached fixture output from live analysis of their own uploaded data before they begin presenting.

**Measurement:** Code — inspect `app.py` source for `st.warning` call guarded by `api_fallback_active`; integration test via Streamlit's `AppTest` harness to assert the warning element is present in the rendered component tree when the flag is set.

**Note on Human review:** The sales engineer role (Section 1b) should run one live demo scenario per sprint to confirm the banner is visible from a projector/screen-share perspective, not just in a local browser. Banner contrast and font size matter for demo room visibility and cannot be verified by code alone.

#### Dimension 3: Fixture Schema Compliance

**What it measures:** `_load_fixture()` returns an `AnalysisResult` instance that passes `model_validate()` against the current schema. Every `CampaignAction` in `fixture_results.json` has all required fields with values that satisfy the Pydantic field constraints (`confidence` in [0.0, 1.0], `percentage_change` in [-100, 100], `budget_action` is a valid Literal value, `source_platforms` is a non-empty list).

**PASS:** `AnalysisResult.model_validate(json.loads(fixture_path.read_text()))` completes without raising `pydantic.ValidationError`. The returned object has `len(campaigns) >= 3` (enough campaigns to demonstrate filtering and comparison features). Each campaign has `evidence_count >= 1` (fixture must not contain campaigns with zero evidence, as those would always return `insufficient_data` and make the decisiveness dimension untestable).

**FAIL:** `model_validate()` raises `ValidationError` (schema drift after a new required field was added without updating the fixture), or the fixture has only 1 campaign (insufficient to demonstrate comparison features), or every campaign has `budget_action == "insufficient_data"` (fixture does not exercise the live recommendation paths).

**Measurement:** Code — `test_fixture_validates()` in `tests/test_llm.py`. Must run on every CI commit that touches `llm.py`, `fixture_results.json`, or the Pydantic schema. Failing this test blocks merge.

```python
def test_fixture_validates():
    from llm import _load_fixture, AnalysisResult
    result = _load_fixture()
    assert isinstance(result, AnalysisResult)
    assert len(result.campaigns) >= 3
    actions = {c.budget_action for c in result.campaigns}
    assert "insufficient_data" not in actions or len(actions) > 1  # not all insufficient
    for c in result.campaigns:
        assert c.evidence_count >= 1
        assert 0.0 <= c.confidence <= 1.0
        assert len(c.source_platforms) >= 1
```

#### Dimension 4: Session State Restoration Fidelity

**What it measures:** After `st_db.load_analysis(run_id)` completes and `st.rerun()` fires, all three session state keys are present with correct types and correct data shapes. The restored state is functionally equivalent to what was set immediately after a fresh analysis run: charts render without error, platform filters populate with the correct campaign list, and PDF/CSV export produces non-empty output.

**PASS:** `st.session_state["analysis_result"]` is an `AnalysisResult` instance (not a raw dict). `st.session_state["campaign_agg"]` is a `pd.DataFrame` with numeric dtype for `spend`, `percentage_change`, and `confidence` columns (not `object` dtype). `st.session_state["merged_df"]` is a `pd.DataFrame` with the same column set as when it was saved. The campaign IDs in `analysis_result.campaigns` match the `campaign_id` values in `campaign_agg`. Navigating to the Charts tab after reload does not raise `KeyError` or `TypeError`. The platform filter dropdown shows options derived from the reloaded data, not from the previous session's data.

**FAIL:** `campaign_agg["spend"]` column dtype is `object` (string) after reload because `pd.DataFrame(records)` inferred the column as string from JSONB serialization. Charts that call `campaign_agg["spend"].sum()` raise `TypeError: unsupported operand type(s) for +: 'int' and 'str'`. Filter options show campaigns from the previous session's data rather than the loaded analysis. PDF export generates a 0-byte file.

**Measurement:** Code — pytest with a real Postgres test database (or `pytest-postgresql` fixture). Save a known `AnalysisResult` + two DataFrames, load them back, assert column dtypes, assert `AnalysisResult` instance type, assert campaign ID match between result and DataFrame.

```python
def test_reload_restores_session_state(pg_conn, sample_payload):
    run_id = st_db.save_analysis("test run", sample_payload, conn=pg_conn)
    payload = st_db.load_analysis(run_id, conn=pg_conn)
    result = AnalysisResult.model_validate(payload["analysis_result"])
    campaign_agg = pd.DataFrame(payload["campaign_agg"])
    assert campaign_agg["spend"].dtype != object  # must be numeric
    saved_ids = {c.campaign_id for c in result.campaigns}
    loaded_ids = set(campaign_agg["campaign_id"].tolist())
    assert saved_ids == loaded_ids
```

#### Dimension 5: Postgres Round-Trip Integrity

**What it measures:** Numeric values for `percentage_change`, `confidence`, and `spend` fields are identical before and after a Postgres JSONB save/load cycle. String values (`campaign_id`, `semantic_reasoning`, `budget_action`) are byte-identical. No columns are added, dropped, or renamed. The `campaigns` list order in `AnalysisResult` is preserved.

**PASS:** `original_result.model_dump() == reloaded_result.model_dump()` (dict equality, not just Pydantic equality). `original_campaign_agg.to_dict(orient="records") == reloaded_campaign_agg.to_dict(orient="records")` after explicit dtype casting on reload. A `percentage_change` value of `25` saved as a Python `int` reloads as `25` (not `25.0` or `"25"`).

**FAIL:** `percentage_change` was saved as `int(25)` but reloads as `float(25.0)` due to JSONB numeric type normalization, causing `original == reloaded` to fail with strict type checking. `semantic_reasoning` strings containing Unicode quotation marks ("budget approved Q3") reload with different encoding. A marketer who saved "increase 25%" presents a reloaded analysis showing "increase 25.0%" — a minor but visible discrepancy that erodes trust in the persistence feature.

**Measurement:** Code — pytest parametrized over all three payload objects with exact value assertion. Use `json.dumps(original) == json.dumps(reloaded)` as the canonical test (JSON serialization normalizes float vs int differences at the boundary where Postgres stores them).

#### Dimension 6: Recommendation Traceability

**What it measures:** The `semantic_reasoning` field for each `CampaignAction` contains specific language drawn from the CRM `sales_notes` column for that campaign — a noun phrase, quoted fragment, or paraphrase that a sales rep reviewing their own notes would recognize. Generic reasoning that could apply to any campaign regardless of input data is a FAIL.

**PASS:** For a campaign where the CRM notes contain "3 of 4 leads mentioned 'budget approved Q3'", the `semantic_reasoning` reads something like "Three leads from Campaign X confirmed Q3 budget approval, supporting a confident increase." The phrase "budget approved Q3" or a paraphrase of it appears in the reasoning. A senior B2B performance marketer reviewing the output can point to the specific note that drove each recommendation.

**FAIL:** `semantic_reasoning` reads "Campaign X shows strong positive signals across lead quality metrics." No CRM language is cited. The same reasoning string could be generated for any campaign regardless of what the sales notes actually said. A sales rep reviewing the output cannot verify whether the AI used their notes or ignored them.

**Measurement:** LLM Judge — evaluate a sample of 5 `semantic_reasoning` strings against the corresponding `sales_notes` input. The judge prompt asks: "Does this reasoning cite specific language from the sales notes provided, or is it generic? Score 1 (generic, no CRM reference) to 5 (specific CRM language clearly cited)." A score of 4 or 5 is PASS.

Calibration requirement: Before trusting the LLM judge in CI, have a senior B2B performance marketer independently score the same 5 examples. Target >= 0.7 Spearman correlation between human scores and judge scores before automating. If the judge scores 3 examples as "specific" that the human scores as "generic," the judge rubric needs refinement.

**Tool:** Run the LLM judge manually during development (not in CI for the hackathon MVP). For the SaaS v2 eval pipeline, integrate with Promptfoo using the `llm-rubric` assertion type.

```yaml
# promptfoo.yaml (reference — not required for hackathon MVP)
providers:
  - openai:gpt-4o-mini  # cheaper model for judging
prompts:
  - "Does this reasoning cite specific language from the sales notes? Score 1-5.\nReasoning: {{reasoning}}\nSales notes: {{notes}}"
tests:
  - vars:
      reasoning: "{{campaign.semantic_reasoning}}"
      notes: "{{campaign.sales_notes}}"
    assert:
      - type: llm-rubric
        value: "Score must be 4 or 5 to pass"
```

#### Dimension 7: Budget Action Decisiveness

**What it measures:** For every `CampaignAction` where `evidence_count > 0` (i.e., at least one CRM-attributed session was present for this campaign), `budget_action` is one of `increase`, `pause`, or `decrease` — not `insufficient_data`. `percentage_change` is non-zero. `confidence` is >= 0.5.

**PASS:** In a dataset with 5 campaigns where all 5 have `evidence_count >= 1`, all 5 `CampaignAction` entries have `budget_action in {"increase", "pause", "decrease"}` and `percentage_change != 0` and `confidence >= 0.5`. The marketer receives a directional recommendation for every campaign they uploaded data for.

**FAIL:** 3 of 5 campaigns with non-zero `evidence_count` return `insufficient_data` with `percentage_change: 0`. The marketer must manually re-examine the same data the tool already received. The tool has provided no decision value for 60% of their campaigns.

**Measurement:** Code — assert on the `AnalysisResult` returned by `run_analysis()` in integration tests with a known `campaign_agg` fixture that has non-zero `evidence_count` for all campaigns.

## 6. Guardrails

### Online (Real-Time)

Online guardrails run on every analysis request. Only one guardrail is warranted here: the fallback banner check. Adding more guardrails would add latency to a single synchronous Streamlit call with no benefit for this system type.

| Guardrail | Trigger | Intervention | Latency Impact |
|-----------|---------|--------------|----------------|
| Fallback banner enforcement | `api_fallback_active=True` is set in session state at the end of `run_analysis()` | Flag: `app.py` renders `st.warning("OpenAI API unavailable — showing cached demo results")` unconditionally before any campaign output is shown. Banner persists until the user triggers a new "Run Analysis" click, which resets the flag to `False`. | Zero — flag check is a dict lookup in Streamlit session state; no additional latency |

**Why only one online guardrail:** The fallback transparency failure mode (Section 1b) is the only one that can cause irreversible trust damage during a live demo. The other failure modes (schema compliance, session restoration) either crash visibly (and are therefore self-reporting) or affect an already-completed action (saving/loading) rather than a real-time inference. A crashed app is better than a silently wrong app for trust — the marketer knows something went wrong and does not act on the output.

**What the guardrail does NOT cover:** The guardrail does not validate the content of the fixture output (that fixture data cannot be mistaken for a specific marketer's live campaigns). This is a UX concern addressed by the fixture design requirement: fixture campaign IDs must use clearly synthetic names (e.g., `DEMO_CAMPAIGN_001`, not `Google Brand — exact match`) so domain mismatch is obvious if noticed.

### Offline (Flywheel)

Offline metrics run on sampled batches after the fact. They feed improvement actions, not real-time interventions.

| Metric | Sampling Strategy | Degradation Signal | Action on Degradation |
|--------|------------------|--------------------|-----------------------|
| Fixture drift rate | Run `test_fixture_validates()` automatically on every commit to `llm.py` or `fixture_results.json` via CI. 100% coverage — not sampled. | `ValidationError` raised by `model_validate()` on the fixture. | Block the commit. Developer must update `fixture_results.json` to match the new schema before merge. |

Check whether recent CRM note formats have diverged from what the prompt was designed for. |
| Fallback rate | Aggregate `api_fallback_active` field in `analysis_logs` weekly. ## 7. Production Monitoring

### Tracing Tool

**Selected: loguru structured logging + Postgres `analysis_logs` table**

Arize Phoenix was evaluated and ruled out (see Section 5, Eval Tooling). For a Streamlit hackathon MVP with one LLM call per session, the `analysis_logs` table achieves equivalent observability at zero additional infrastructure cost and no new dependencies.

The `analysis_logs` table is created alongside `analysis_runs` in `st_db.init_db()`:

```sql
CREATE TABLE IF NOT EXISTS analysis_logs (
    id           SERIAL PRIMARY KEY,
    logged_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    run_mode     TEXT NOT NULL,          -- 'live' | 'fixture' | 'demo_mode'
    error_type   TEXT,                   -- openai exception class name, NULL if live succeeded
    campaign_count INT,                  -- number of campaigns in the analysis
    latency_ms   INT,                    -- wall-clock ms for the full run_analysis() call
    api_fallback_active BOOLEAN NOT NULL DEFAULT FALSE
);
```

Write a log entry at the end of every `run_analysis()` call:

```python
# llm.py — after the try/except block resolves
import time
_log_analysis_run(
    run_mode="fixture" if fallback_fired else "live",
    error_type=exc_type_name,   # None if live succeeded
    campaign_count=len(result.campaigns),
    latency_ms=int((time.perf_counter() - t0) * 1000),
    api_fallback_active=fallback_fired,
)
```

loguru handles structured logging for the VPS stderr/file log; the `analysis_logs` table handles queryable aggregations.

---

### Key Metrics to Track

| Metric | Query | Normal Range | Notes |
|--------|-------|--------------|-------|
| Fallback rate | `SELECT COUNT(*) FILTER (WHERE api_fallback_active) / COUNT(*)::float FROM analysis_logs WHERE logged_at > NOW() - INTERVAL '7 days'` | < 5% | Elevated rate indicates API key expiry, rate limit tier issue, or network instability |
| Live vs. fixture split | `SELECT run_mode, COUNT(*) FROM analysis_logs GROUP BY run_mode` | > 90% live in normal operation | High fixture % during demos is expected; persistent high fixture % outside demo hours signals a problem |
| Reload success rate | `SELECT COUNT(*) FROM analysis_logs WHERE run_mode = 'reload' AND error_type IS NULL` / total reload attempts | > 99% | Any non-zero reload failure rate should be investigated — Postgres is the only dependency |
| Median analysis latency | `SELECT PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY latency_ms) FROM analysis_logs WHERE run_mode = 'live'` | 3,000–8,000 ms | P50 > 15,000 ms suggests rate limiting or API slowdown; P50 < 500 ms is unexpected and may indicate the live path was skipped |
| Fixture schema drift events | CI test failure on `test_fixture_validates` | 0 per sprint | Any non-zero count means a schema change was made without updating the fixture |

---

### Alert Thresholds

| Condition | Threshold | Response |
|-----------|-----------|----------|
| Fallback rate spike | > 30% of analysis runs in a rolling 1-hour window | Check OpenAI status page; verify `OPENAI_API_KEY` validity; check VPS network connectivity to `api.openai.com` |
| Analysis latency spike | P95 latency > 30,000 ms for 3 consecutive runs | Check for oversized `campaign_agg` payload approaching the 128k token window; verify tenacity retry count is not exhausting the timeout budget |
| Reload failure | Any `st_db.load_analysis()` raising `psycopg2.Error` or `pydantic.ValidationError` | Check Postgres connection; verify `DATABASE_URL` env var; run `test_fixture_validates()` to confirm schema drift is not the cause |
| Fixture drift | `test_fixture_validates()` fails in CI | Block merge — developer must update `fixture_results.json` before the commit can land |

For the hackathon MVP, "alerting" means checking the `analysis_logs` table and loguru output before a demo session. No automated alerting infrastructure (PagerDuty, Sentry) is needed at this stage. For SaaS v2, add Sentry for uncaught exceptions and a cron job that queries `analysis_logs` daily.

### Smart Sampling Strategy

Sampling weights toward the interactions most likely to reveal problems. For the `analysis_logs` table, queries can be filtered and ordered to surface high-signal rows first.

| Sampling Weight | Filter | Rationale |
|-----------------|--------|-----------|
| Highest priority | `api_fallback_active = TRUE` | Every fallback event deserves review during the hackathon phase — these are the trust-damaging failure mode |
| High priority | `latency_ms > 15000` | Slow live calls indicate rate limiting or oversized payloads — review before the next demo session |
| High priority | `error_type IS NOT NULL AND run_mode != 'fixture'` | Unexpected exception types outside the expected fallback path indicate code bugs |
| Medium priority | `run_mode = 'live' ORDER BY RANDOM() LIMIT 5` per week | Random sample of live recommendations for traceability LLM judge review |
| Low priority | `run_mode = 'fixture' AND api_fallback_active = FALSE` | Explicit DEMO_MODE runs — expected behavior, only review if fixture content concerns arise |

**Human review cadence:** Before each demo session, run:

```sql
-- Pre-demo health check: any anomalies in last 24h?
SELECT run_mode, error_type, COUNT(*), AVG(latency_ms)
FROM analysis_logs
WHERE logged_at > NOW() - INTERVAL '24 hours'
GROUP BY run_mode, error_type
ORDER BY COUNT(*) DESC;
```

Any `error_type IS NOT NULL` rows where `run_mode != 'fixture'` are unexpected and should be investigated before the demo.
